怪异的梦

不知是这几天头晕，还是严重缺失睡眠，昨晚做了一个异常怪异的梦。
梦中，我打一群人在看热闹，但是也不知道看什么，只是知道一群人围观在那里，不到一伙，人群散了，我也离开，突然有人抓我的腿，还有人抓我的手，似乎要把我抬起来，他们口里还说“抓回去，抓回去”
我慌忙问，“你们抓我做什么，我又不认识你们？”

那群人却说：“你是回族人，要抓你回去，这里是汉族住的地方”
我就拼命解释我不是回族人，我是汉族人。我不会你们回去，他们就追我，我就拼命跑，一直跑到大家都累了，都跑不动了，我也突然醒来了，一摸额头，湿的，身上也出汗了。
早晨一直在想怎么会这样呢？为什么会梦到这样奇怪的事情呢？常说，“日有所思，也有所梦”，可我压根就没有想到什么回族之类的呀。Rachel说我是看了《狼图腾》的原因，这也是很早就看完了，而这段时间我一直在看《废都》，昨晚睡觉前也看了一段，但是《废都》里面压根就没有说道回族呀，又不是看《穆斯林的葬礼》。
难道是《废都》后面会说道回族，看情节也不太可能呀，于是趁中午休息的时间，看完了最后 4 章，依然没有说道回族，这就怪异了。

而唯一与这有点点关联的是有很多人第一次见到我，都要问我哪里人，然后就说不想，说我应该是新疆的啦，或者是什么哈萨克斯坦人啦，反正应该是西域一代的，如果不是，要不就是你们家祖上有西域的，吓，这我还真不知道了，活血有机会去了解一下。

感叹爱情－－写给freecloudz

很长时间没有和 freecloudz 联系了，今日在 MSN 上见到他，居然告诉我他和女友分手了。又给我看了他在天涯写的《真的很痛》，看了很长时间，唏嘘不已。

他女友我是认识的，比我们低一届。严格说来，他们认识还是我介绍的，当时只是说我介绍一下，剩下的就靠你了。
从那以后，freecloudz 锲而不舍，依然记得圣诞节的时候，freecloudz 冒着被逮住的危险，翻阅围墙送玫瑰的情形，依然记得他总会变着法子来过他们的认识一周纪念，认识一月纪念等。多年来，他对他的付出，不仅仅让她感动，也让她的朋友感动和我们感动。也许正式这种锲而不舍，他们终于走到了一起，真的成了男女朋友。毕业后，他留下了陪他靠 GMAT，之后又离开长沙，去深圳，也是为了她。
之间种种努力，也时常使得她泪流满面，觉得她今生再也早不到比他跟爱她的人了。
可惜感动依然只是感动，物质的诱惑总是那样的让人难以抗拒，3 个月的物质诱惑居然冲垮了 6 年来建立的爱情，再爱情和面包，或者面包和海鲜面前，后者总是会以胜利的姿态出现。

又或许感动并不是爱情，她也许一直生活在感动中，误以为感动便是爱情，而这种感动总是一方真心的付出，一方只需要享受，爱情应该生活平等的，她不是一方的付出，而是两人的相互付出和回报。如果真的爱过，6 年的感情不是说消失就消失的，也许有时分手才能真的看出双方是否真的深爱着对方，那么对分手后又复合的，那是因为走过之后，才发现原来的她（他）才是之间的真爱。

freecloudz，守候你的爱情吧。保重。

device-mapper-multipath的灵异事件--3

来深圳出差是解决 openpower710+redflag 连接 EMC
CX300 存储的故障问题，而这个连接方案我已经在北京作的测试，也有对应的测试报告，但是同样的环境，在深圳却找不到存储设备，很是怪异。而最后的解决办法更是让我搞不明白，因此标题冠以“灵异”二字，而具体的情况估计也说不太清楚，就贴上我的当时的一些记录吧，这是其三

将前两天配置的机器重新安装，重新调试 multipath，数据读写都没有任何问题，估计昨天出错的原因是原来的 VG 是在一个控制器使用下的情况配置的，导致现在双控制器下读写不正常。但是 multipath 却出现了路径不能”愈合”的现象，既拔掉一根光纤线后，一个路径失效 (假设 sde), 然后重新介入，发现该路径依然无效。同时使用 multipath -v3 时还是会给出下面的错误：

path not correctly configured for failover

该信息很难在网络上找到对应的资源。
而 multipathd 服务也会在路径失效的情况下异常退出。而该 daemon 程序就是用来检查路径状态和恢复链路的服务程序。
重启 multipathd 服务后，会给出下面的错误：

device-mapper: dm-multipath: Failing path 8:48.
device-mapper: dm-multipath: Failing path 8:80.
device-mapper: dm-multipath: Failing path 8:32.
device-mapper: dm-multipath: Failing path 8:64.

从这个错误信息来看，multipathd 并不认为路径状态是好的，虽然你可以正常读写每一条路径。因此使用 multipath -ll 看到下面的信息

[root@SZFIMSDB ~]# multipath -ll
3600601609f70140076916a71f81edb11
[size=66 GB][features="1 queue_if_no_path"][hwhandler="1 emc"]
\_round-robin 0 [enabled]
\_9:0:0:0 sdc 8:32 [faulty][active]
\_ 10:0:0:0 sde 8:64 [faulty][active]
\_ round-robin 0 [enabled]
\_ 9:0:1:0 sdd 8:48 [faulty][active]
\_ 10:0:1:0 sdf 8:80 [faulty][active]

后面的[faulty]是表示路径的状态，faulty 表示当前无法处理 I／0 请求。这应该是不能路径不能愈合的原因。
无数电话和研发沟通后，研发认为是工具在误报，可以考虑换一个 multipath 版本测试或者重新编译新的 multipath。

测试了 DC5.0 sp1 自带的 0.4.5-12 的版本和研发先前给我的 0.4.5-6 的版本，还是有这个错误。
修改过无数种 multipath 的配置，仍然无效。

久违的感觉

wirlfy送mswishes上火车后，直接来了这里。

一顿豪华版晚餐后，开始了我几天指定的主题－－讨论网络存储的相关内容。
我想这周五能和同事做交流。无赖对于存储的关注已经丢失了很久，需要 wirlfly 的帮助，他给我将了 CAS 的一些相关理论，于是我们有 CAS 讨论了 EMC 的 centera，讨论到了 CASTOR，讨论了 NASD, 讨论了 OBS，讨论了 SAN 存储虚拟化。

似乎两个人都热衷于这样的讨论，不知不觉已到一点，大家还是没有睡意。不过考虑明天要上班，只能作罢。这种激烈的讨论的结果是一晚没有睡好，一直在做梦，而做梦的内容居然全部是什么 NAS，SAN，iSCSI,IP SAN 等网络存储的东西，果然是日有所思，夜有所梦呀。

很喜欢这种讨论的感觉，今晚是不是又回去讨论呢？

grub提示符下安装grub

一直在找如何在 grub 丢失，只剩下 grub> 的情况下，直接安装 grub 菜单，而不是输入 root,kernel,initrd 一堆命令后进入系统，再使用 grub-install 来安装 grub。
今天Rachel从她的笔记本上找到了这条命令，果然管用，记录如下：
grub>install (hd0,4)/boot/grub/stage1 (hd0) (hd0,4)/boot/grub/stage2 p
(hd0,4)/boot/grub/grub.conf
注意：
这里表示根文件系统是 hda5，同时 boot 没有单独分出来。如果你的 boot 是单独分出来的，那么“(hd0,4)/boot”两处就要修改成你的 boot 分区了。
