Device-mapper-multipath的灵异事件--3

| Comments

来深圳出差是解决openpower710+redflag连接EMC CX300存储的故障问题，而这个连接方案我已经在北京作的测试，也有对应的测试报告，但是同样的环境，在深圳却找不到存储设备，很是怪异。而最后的解决办法更是让我搞不明白，因此标题冠以“灵异”二字，而具体的情况估计也说不太清楚，就贴上我的当时的一些记录吧，这是其三

将前两天配置的机器重新安装，重新调试multipath，数据读写都没有任何问题，估计昨天出错的原因是原来的VG是在一个控制器使用下的情况配置的，导致现在双控制器下读写不正常。但是multipath却出现了路径不能”愈合”的现象，既拔掉一根光纤线后，一个路径失效(假设sde),然后重新介入，发现该路径依然无效。同时使用multipath -v3时还是会给出下面的错误：

path not correctly configured for failover

该信息很难在网络上找到对应的资源。 而multipathd服务也会在路径失效的情况下异常退出。而该daemon程序就是用来检查路径状态和恢复链路的服务程序。 重启multipathd服务后，会给出下面的错误：

device-mapper: dm-multipath: Failing path 8:48.
device-mapper: dm-multipath: Failing path 8:80.
device-mapper: dm-multipath: Failing path 8:32.
device-mapper: dm-multipath: Failing path 8:64.

从这个错误信息来看，multipathd并不认为路径状态是好的，虽然你可以正常读写每一条路径。因此使用multipath -ll看到下面的信息

[root@SZFIMSDB ~]# multipath -ll
3600601609f70140076916a71f81edb11
[size=66 GB][features="1 queue_if_no_path"][hwhandler="1 emc"]
\_round-robin 0 [enabled]
\_9:0:0:0 sdc 8:32 [faulty][active]
\_ 10:0:0:0 sde 8:64 [faulty][active]
\_ round-robin 0 [enabled]
\_ 9:0:1:0 sdd 8:48 [faulty][active]
\_ 10:0:1:0 sdf 8:80 [faulty][active]

后面的[faulty]是表示路径的状态，faulty表示当前无法处理I／0请求。这应该是不能路径不能愈合的原因。 无数电话和研发沟通后，研发认为是工具在误报，可以考虑换一个multipath版本测试或者重新编译新的multipath。

测试了DC5.0 sp1自带的0.4.5-12的版本和研发先前给我的0.4.5-6的版本，还是有这个错误。
修改过无数种multipath的配置，仍然无效。
