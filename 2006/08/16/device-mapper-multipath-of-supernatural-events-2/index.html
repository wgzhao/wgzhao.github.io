device-mapper-multipath的灵异事件--2

来深圳出差是解决 openpower710+redflag 连接 EMC
CX300 存储的故障问题，而这个连接方案我已经在北京作的测试，也有对应的测试报告，但是同样的环境，在深圳却找不到存储设备，很是怪异。而最后的解决办法更是让我搞不明白，因此标题冠以“灵异”二字，而具体的情况估计也说不太清楚，就贴上我的当时的一些记录吧，这是其二

今天上午 EMC 的人过来了，只是看了一下 CX300 的状态，说从状态来看，没有问题。
然后就走了。

雁联的工程师继续安装 powerpath for AIX。我和 hyphwang 继续研究 CX300 与 linux 相关的配置。

后来在 AIX 上配置成了 powerpath，发现 SPA，SPB 均是有效状态。
但是由于现在的 LUN 上已经有数据，于是没有进一步去做卷管理方面的操作。

我们设置了 CX300 上一个 ArrayCommPath 开关，由原来的 enable 设置为 disable。
同时将 failover 改为 2。
在 linux 系统上，能成功认出 4 个设备，且大小一样。这样就解决了昨天不能发现四
个大小一致的设备的问题。
接着做 multipath，会给出下面的错误：

path not correctly configured for failoverpath not correctly configured for failover

经和研发无数电话沟通后，认为可能是 multibus 和 failover 功能选择问题。研发给
了一个新的multipath.conf配置文件。

但是还是有一些问题。但是上面的 VG 信息都是正确的，也能挂载 LV。看上去一切正
常。也能 copy 数据，但是在作 rm -rf 命令时，dm-emc模块立刻给出 trespass 命令。
系统无法正常关机。

经过若干调试，结果依然。

晚上 9 点，决定重做 VG，实际上现在的 VG 也是读不出来。一读就和执行 rm 同样的现象。

既然重做 VG，于是考虑在 AIX 测试 copy 命令。结果一切正常。

接着在 Linux 系统上测试，先分别对 sdc,sdd,sde,sdf 做
dd if=/dev/zero of=/dev/sdx bs=1M count=2048
的操作，同时使用 iostat 查看 IO，一切正常。

启动 multipath，对/dev/dm-2也执行 dd 命令，一切正常。

pvcreate,vgcreate,lvcreate，一些正常。
mkfs,mount, 一切正常。

copy，一些正常。
rm -rf , 出现上面同样的错误。

电话和研发沟通，他认为系统可能不稳定，考虑重装。
同时查看了 device-mapper 的版本和架构，发现是 32bit 的。
而 device-mapper-multipath 是 64bit 的，显然 device-mapper-multipath 需要依赖
device-mapper 库的。

查看 DC5.0 for pSeries 第一张安装光盘，device-mapper 有两个包。

device-mapper-1.01.01-1.RHEL4.ppc64.rpm
device-mapper-1.01.01-1.RHEL4.ppc.rpm

系统默认的是 ppc.rpm
查看系统其他一些软件，发现都是 32bit 的，但是使用
uname -a 看系统内核，发现是 ppc64 的 再看光盘的 RPM 包，很多软件同时具有 ppc.rpm 和 ppc64.rpm，但是系统默认是安装
ppc.rpm，也就是 32bit 的，不解？

去查看另外一台没有怎么动，算是比较干净的系统，发现绝大部分软件都是
32bit，不知何解？

明天计划：

  1. 这两天折腾的机器，重装系统。
  2. 使用同结构（32bit，64bit）的 device-mapper,device-mapper-multipath 再测试。
  3. 同时再去看看 CX300 是不是还有可以调试的地方，他很多参数的设置对 Linux 有影响，更具体的说是对 Emulex 光纤卡的驱动有影响。
