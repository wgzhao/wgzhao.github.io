Device-mapper-multipath的灵异事件--2

| Comments

来深圳出差是解决openpower710+redflag连接EMC CX300存储的故障问题，而这个连接方案我已经在北京作的测试，也有对应的测试报告，但是同样的环境，在深圳却找不到存储设备，很是怪异。而最后的解决办法更是让我搞不明白，因此标题冠以“灵异”二字，而具体的情况估计也说不太清楚，就贴上我的当时的一些记录吧，这是其二

今天上午EMC的人过来了，只是看了一下CX300的状态，说从状态来看，没有问题。 然后就走了。

雁联的工程师继续安装powerpath for AIX。我和hyphwang继续研究CX300与linux相关的配置。

后来在AIX上配置成了powerpath，发现SPA，SPB均是有效状态。 但是由于现在的LUN上已经有数据，于是没有进一步去做卷管理方面的操作。

我们设置了CX300上一个ArrayCommPath开关，由原来的enable设置为disable。 同时将failover改为2。 在linux系统上，能成功认出4个设备，且大小一样。这样就解决了昨天不能发现四 个大小一致的设备的问题。 接着做multipath，会给出下面的错误：

path not correctly configured for failoverpath not correctly configured for failover

经和研发无数电话沟通后，认为可能是multibus和failover功能选择问题。研发给 了一个新的multipath.conf配置文件。

但是还是有一些问题。但是上面的VG信息都是正确的，也能挂载LV。看上去一切正 常。也能copy数据，但是在作rm -rf 命令时，dm-emc模块立刻给出trespass命令。 系统无法正常关机。

经过若干调试，结果依然。

晚上9点，决定重做VG，实际上现在的VG也是读不出来。一读就和执行rm同样的现象。

既然重做VG，于是考虑在AIX测试copy命令。结果一切正常。

接着在Linux系统上测试，先分别对sdc,sdd,sde,sdf做 dd if=/dev/zero of=/dev/sdx bs=1M count=2048 的操作，同时使用iostat查看IO，一切正常。

启动multipath，对/dev/dm-2也执行dd命令，一切正常。

pvcreate,vgcreate,lvcreate，一些正常。 mkfs,mount,一切正常。

copy，一些正常。 rm -rf ,出现上面同样的错误。

电话和研发沟通，他认为系统可能不稳定，考虑重装。
同时查看了device-mapper的版本和架构，发现是32bit的。 而device-mapper-multipath是64bit的，显然device-mapper-multipath需要依赖 device-mapper库的。

查看DC5.0 for pSeries第一张安装光盘，device-mapper有两个包。

device-mapper-1.01.01-1.RHEL4.ppc64.rpm
device-mapper-1.01.01-1.RHEL4.ppc.rpm

系统默认的是ppc.rpm 查看系统其他一些软件，发现都是32bit的，但是使用 uname -a看系统内核，发现是 ppc64的 再看光盘的RPM包，很多软件同时具有ppc.rpm和ppc64.rpm，但是系统默认是安装 ppc.rpm，也就是32bit的，不解？

去查看另外一台没有怎么动，算是比较干净的系统，发现绝大部分软件都是 32bit，不知何解？

明天计划：

  1. 这两天折腾的机器，重装系统。
  2. 使用同结构（32bit，64bit）的device-mapper,device-mapper-multipath再测试。
  3. 同时再去看看CX300是不是还有可以调试的地方，他很多参数的设置对Linux有影响，更具体的说是对Emulex光纤卡的驱动有影响。
